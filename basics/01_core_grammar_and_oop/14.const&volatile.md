## [C++ Concept] The `volatile` Keyword

### 1. Definition
The `volatile` keyword tells the compiler that a variable's value can be changed by factors **outside the program's control** (e.g., hardware, OS, or another thread). It strictly forbids the compiler from applying "caching" optimizations on that variable.

### 2. Why do we need it?
Without `volatile`, the compiler might assume a variable doesn't change if the code doesn't explicitly modify it. It might then keep the value in a **CPU Register** instead of checking **RAM** every time, leading to infinite loops when the value actually changes externally.



### 3. Code Example: Hardware Polling
```cpp
#include <iostream>

int main() {
    // 0x4000 represents a hardware register (e.g., a physical button)
    // We use 'volatile' to ensure we read the ACTUAL memory address every time.
    volatile int* status_reg = reinterpret_cast<volatile int*>(0x4000);

    std::cout << "Waiting for hardware signal..." << std::endl;

    // Without volatile: Compiler might turn this into 'while(true)' if it thinks *status_reg is 0
    // With volatile: Compiler generates code to load from memory 0x4000 on every iteration
    while (*status_reg == 0) {
        // Busy-wait (Polling)
    }

    std::cout << "Signal received! Proceeding..." << std::endl;
    return 0;
}
```

### 4. Key Rules for your Notes
* **Type Safety:** `volatile int*` cannot be cast to `void*` using `static_cast`. It must be cast to `volatile void*`.
* **Not for Thread Safety:** `volatile` is **not** a substitute for `std::atomic` or `std::mutex`. It ensures memory visibility, but it does **not** prevent "Race Conditions" or ensure "Atomicity."
* **The "CV" Duo:** `const` and `volatile` are both "Type Qualifiers."
    * `const`: "I promise not to change this."
    * `volatile`: "Someone else might change this."

## [C++ Concept] The `constexpr` Keyword

### 1. Definition
`constexpr` tells the compiler that a value or function **can be evaluated at compile time**. When all inputs are known at compile time, the result is computed during compilation and baked into the binary as a constant — zero runtime cost.

### 2. `const` vs `constexpr`
* `const`: "This value will not change after initialization." It may still be initialized at **runtime**.
* `constexpr`: "This value is a **compile-time constant**." It is stricter — every `constexpr` variable is implicitly `const`, but not every `const` is `constexpr`.

```cpp
int runtime_value();

const int a = runtime_value();      // OK: const, but computed at runtime
constexpr int b = 5 * 4;            // OK: known at compile time
// constexpr int c = runtime_value(); // ERROR: not a compile-time constant
```

### 3. `constexpr` Functions
A `constexpr` function can run at compile time **or** runtime, depending on whether its arguments are compile-time constants.

```cpp
constexpr int factorial(int n) {
    return (n <= 1) ? 1 : n * factorial(n - 1);
}

int main() {
    constexpr int x = factorial(5);  // computed at compile time -> 120
    int n = 5;
    int y = factorial(n);            // computed at runtime (n is not constexpr)
    return x + y;
}
```

A common use is fixed-size arrays, where the size must be a compile-time constant:

```cpp
constexpr int size = factorial(4); // 24
int buffer[size];                   // legal: size is a compile-time constant
```
위의 예시가 효과적인 이유는 안좋은 예시를 보면 알 수 있음:
```cpp
int n;
std::cin >> n;
int buffer[n]; // error: 크기를 모르니까 메모리 칸을 못 짬.
```

### 4. Related: `consteval` and `constinit` (C++20)
* `consteval`: an "immediate function" that **must** run at compile time. Calling it with runtime values is an error.
* `constinit`: guarantees a variable is **initialized at compile time** (avoids the static initialization order problem), but unlike `constexpr` it does **not** make the variable `const`.

```cpp
consteval int square(int n) { return n * n; }
constexpr int s = square(8);   // OK
// int r = square(runtime());  // ERROR: must be a constant expression

constinit int counter = square(3); // compile-time init, but still mutable
```

### 5. Key Rules for your Notes
* **`constexpr` implies `const`** for variables (but not the other way around).
* **Use it for:** array sizes, template arguments, lookup tables, and anything you want resolved before the program runs.
* **`if constexpr`** (covered in templates) picks a branch at compile time and discards the other — great for generic code.
* **Mental model:**
    * `const`: "won't change."
    * `constexpr`: "known at compile time."
    * `consteval`: "**must** be computed at compile time."
    * `constinit`: "**initialized** at compile time (still mutable)."
