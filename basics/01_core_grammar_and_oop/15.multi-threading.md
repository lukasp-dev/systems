# Multi-threading

C++ 에는 Multithreading 을 구현하는 방법이 여러가지가 있는데 어떤게 있는지 잘 알아보자.

### 1) `std:thread`
- All parameters are passed by **value** unless used `std::ref` to pass by **reference**.
```cpp
#include <iostream>
#include <vector>      // std::vector 사용을 위해 필요
#include <thread>      // std::thread 사용을 위해 필요
#include <numeric>     // std::accumulate 사용을 위해 필요
#include <cstdint>     // uint64_t 정의를 위해 필요 (uint64_t.h는 없음)

// 1. sum을 참조(&)로 받으므로 호출 시 std::ref가 필수입니다.
void AccumulateRange(uint64_t &sum, uint64_t start, uint64_t end) {
    sum = 0;
    for (uint64_t i = start; i < end; ++i) {
        sum += i;
    }
}

int main() {
    const int number_of_threads = 2;
    // 2. 10억은 int 범위를 넘을 수 있으므로 안전하게 uint64_t로 선언합니다.
    const uint64_t number_of_elements = 1000 * 1000 * 1000;
    const uint64_t step = number_of_elements / number_of_threads;
    
    std::vector<uint64_t> partial_sums(number_of_threads);

    // 3. std::thread 생성 및 실행
    // t2의 끝 범위를 number_of_threads * step 대신 number_of_elements로 써도 무방합니다.
    std::thread t1(AccumulateRange, std::ref(partial_sums[0]), 0, step);
    std::thread t2(AccumulateRange, std::ref(partial_sums[1]), step, number_of_elements);

    // 4. 스레드가 끝날 때까지 대기
    t1.join();
    t2.join();

    // 5. 최종 합산
    uint64_t total = std::accumulate(partial_sums.begin(), partial_sums.end(), uint64_t(0));
    std::cout << "total: " << total << std::endl;

    return 0;
}
```

If the threads are so many,

```cpp
#include <iostream>
#include <vector>
#include <thread>
#include<chrono>
#include <numeric>
#include <iomanip>

using namespace std;
const uint64_t operations = 100'000'000;

void Task(uint64_t& sum, uint64_t count) {
    for(uint64_t i=0; i<count; ++i){
        sum++; // 여기서 계속 메모리에 접근하며 경합 발생
    }
}


void RunTest(int num_threads) {
    vector<uint64_t> partial_sums(num_threads, 0);
    vector<thread> threads;
    uint64_t count_per_thread = operations / num_threads;

    auto start = chrono::high_resolution_clock::now();

    for(int i=0; i < num_threads; i++) {
        threads.emplace_back(Task, ref(partial_sums[i]), count_per_thread);
    }

    for(auto& t : threads) {
        t.join();
    }

    auto end = chrono::high_resolution_clock::now();
    chrono::duration<double, milli> elapsed = end - start;

    cout << "Threads: " << setw(3) << num_threads
    << " | Time: " << setw(10) << elapsed.count() << " ms" << endl;
}

int main() {
    // 1개(순차 실행)부터 128개까지 늘려가며 테스트
    vector<int> thread_counts = {1, 2, 4, 8, 16, 32, 64, 128};
    
    cout << "--- Starting Multi-threading Performance Test ---" << endl;
    for (int count : thread_counts) {
        RunTest(count);
    }
    return 0;
}
```

### 2) using a functor

```cpp
#include <iostream>
#include <vector>
#include <thread>
#include <numeric>
#include <functional> // Required for std::ref

class AccumulateFunctor {
public:
    // Fixed typo: changed t_sum to _sum to match the member name
    AccumulateFunctor() : _sum{0} {}

    void operator()(uint64_t start, uint64_t end) {
        _sum = 0; // Local accumulation
        for (uint64_t i = start; i < end; ++i) {
            _sum += i;
        }
    }

    uint64_t _sum; // Moved to public so main() can read it
};

int main() {
    const int number_of_threads = 10;
    uint64_t number_of_elements = 1000'000'000; 
    uint64_t step = number_of_elements / number_of_threads;

    std::vector<std::thread> threads;
    std::vector<AccumulateFunctor*> functors;

    // Launching threads
    for (int i = 0; i < number_of_threads; ++i) {
        AccumulateFunctor* functor = new AccumulateFunctor();
        functors.push_back(functor);
        
        uint64_t start = i * step;
        // Ensures the last thread covers any remainder
        uint64_t end = (i == number_of_threads - 1) ? number_of_elements : (i + 1) * step;

        // Use std::ref to ensure the thread works on the actual object on the heap
        threads.emplace_back(std::ref(*functor), start, end);
    }

    // Joining threads: Wait for all work to finish
    for (std::thread & t : threads) {
        if (t.joinable()) {
            t.join();
        }
    }

    // Aggregatin g results: The final "Reduction" step
    uint64_t total = 0; 
    for (auto pf : functors) {
        total += pf->_sum;
    }

    std::cout << "Total: " << total << std::endl; // 499999999500000000

    // Manual memory management
    for (auto pf : functors) {
        delete pf;
    }

    return 0;
}
```

### 3) Using lambda functions

```cpp
#include <iostream>
#include <thread>
#include <vector>

int main() {
    int total = 0;
    std::vector<int> data{1,2,3,4,5,6,7,8};

    std::thread t([&] {
        int local = 0;
        for (int x : data) local += x;
        total = local;
    });

    t.join();
    std::cout << "total: " << total << "\n";
}
```

### 4) Synchronization primitives you must know

### `std::mutex` + `std::lock_guard`
- Protect shared mutable state.
- `lock_guard` gives RAII locking and auto-unlock on scope exit.

```cpp
std::mutex m;
int counter = 0;

void inc() {
    std::lock_guard<std::mutex> lock(m);
    ++counter;
}
```

### `std::unique_lock`
- Slightly heavier than `lock_guard`, but supports manual unlock/relock.
- Required by `std::condition_variable::wait`.

### `std::condition_variable`
- One or more threads wait for a condition.
- Always wait with predicate to handle spurious wakeups.

```cpp
std::mutex m;
std::condition_variable cv;
bool ready = false;

void consumer() {
    std::unique_lock<std::mutex> lk(m);
    cv.wait(lk, [] { return ready; });
}
```

### 5) Atomics and memory ordering (high-level)
- `std::atomic<T>` avoids data races for simple shared variables.
- Interview-safe summary:
  - `memory_order_relaxed`: atomicity only, no ordering.
  - `memory_order_release` + `memory_order_acquire`: producer-consumer ordering.
  - `memory_order_seq_cst`: strongest, easiest mental model, usually slower.

### 6) Race condition vs deadlock
- Race condition: incorrect result because unsynchronized interleaving.
- Deadlock: threads wait forever due to circular lock dependency.
- Avoid deadlock by fixed lock ordering or `std::scoped_lock`.

### 7) Modern lifecycle note (`std::jthread`, C++20)
- `std::thread` must be joined/detached manually.
- `std::jthread` auto-joins in destructor and supports cooperative stop tokens.

### 8) Memory model intuition: happens-before

Interview-safe rule:
- If write in thread A must be visible in thread B, establish synchronization edge.
- That edge is called **happens-before**.
- Mutex lock/unlock, condition_variable wait/notify, and release/acquire atomics create this edge.

Without happens-before, reading shared data is either stale or undefined behavior (data race case).

### 9) Producer-consumer with release/acquire

```cpp
#include <atomic>
#include <iostream>
#include <thread>

std::atomic<bool> ready{false};
int payload = 0; // ordinary variable, guarded by ready protocol

void producer() {
    payload = 42; // (1) write data first
    ready.store(true, std::memory_order_release); // (2) publish
}

void consumer() {
    while (!ready.load(std::memory_order_acquire)) {
        // spin
    }
    // acquire sees all writes sequenced-before release in producer
    std::cout << payload << "\n"; // guaranteed 42
}

int main() {
    std::thread t1(producer), t2(consumer);
    t1.join();
    t2.join();
}
```

Why this works:
1. producer's `release` store publishes prior writes (`payload = 42`)
2. consumer's `acquire` load that observes `true` synchronizes-with the release store
3. that establishes happens-before from producer writes to consumer reads

### 10) Wrong vs correct synchronization pattern

Wrong (data race):
```cpp
bool ready2 = false;
int data2 = 0;
// thread A: data2 = 10; ready2 = true;
// thread B: if (ready2) std::cout << data2; // race + no ordering
```

Correct:
- make `ready2` atomic with release/acquire, or
- guard both `ready2` and `data2` with the same mutex.
