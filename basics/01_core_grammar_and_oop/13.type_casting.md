# Type Casting & Memory Interpretation

In low-latency systems, casting is not just about changing types. It is about defining how the CPU interprets a sequence of bits. We categorize by their Runtime Cost and Type Safety.

## type casting 4 대장

### 1. `static_cast`
`static_cast` performs conversions between related types at **compile-time**. It is the go-to for safe, predictable transformations. (does not allow unreasonable type casting like from `int` to `vector<int>`)
- **Mechanism:** It can change the underlying bit representation (e.g., converting an `int` to a `float` follows IEEE 764 conversion rules).
- **Cost**: Zero Runtime overhead. The conversion logic is backed into the instructions (e.g., `CVSI2SD` in x86).

### 2. `reinterpret_cast`
This tells the compiler: "Ignore the type system and treat these bits as this other type."
- Mechanism: It does not change a single bit. It only changes the compiler's "view" of the memory address.
- Use-Case: Mapping a raw network buffer (from a Solarflare or Mellanox NIC) directly onto a `struct` representing an Exchange Order Book Update.
- The Danger: It easily triggers **Strict Aliasing** violations. If you cast `int*` to `float*` and modify it, the compiler's optimizer might assume they are distinct memory locations and produce "Heisenbugs" (bugs that disappear when debugging).
```cpp
#include <bit>
#include <cstdint>

struct marketUpdate {
    unit64_t sequence;
    double price;
    int32_t quantity;
};

void process_raw_bytes(char* buffer){
    // ❌ WRONG: dynamic_cast (will not compile/too slow)
    // ⚠️ RISKY: reinterpret_cast (potential Aliasing issues)
    // ✅ BEST: std::bit_cast (C++20) or memcpy
    auto* update = reinterpret_cast<MarketUpdate*>(buffer);
}
```

### 3. `const_cast`
Used exclusively to add or remove the `const` or `volatile` qualifiers.

### 4. `dynamic_cast`
Used for safe downcasting in polymorphic hierarchies using **RTTI**(Runtime Type Information)
- 만약 변환이 불가한 타입이라면:
    - in case of a pointer: return `nullptr`
    - in case of a reference: throws `std::bad_cast` exception
- Downcasting: 부모 클래스 포인터를 자식 클래스 포인터로 바꿀 때 주로 쓴다.

```cpp
#include <iostream>

class Animal {
public:
    virtual ~Animal() {}
};

class Dog : public Animal { public: void bark() {std::cout << "Woof!\n";}};
class Cat : public Animal { public: void meow() {std::cout << "Meow!\n";}};

void identify(Animal* a) {
    if (Dog* dog = dynamic_cast<Dog*>(a)) {
        dog->bark();
    }
    
    else if (Cat* cat = dynamic_cast<Cat*>(a)) {
        cat->meow();
    }
    
    else {
        std::cout << "Unknown animal\n";
    }
}
```

### Why avoid C-style casts `(int)x`?
They are "blind" casts. They try `static_cast`, `reinterpret_cast`, and `const_cast` in a hidden order. They are hard to search for in a codebase and hide dangerous bugs.

### `bit_cast`: the new feature feature in C++20
Before C++20, if you wanted to look at the underlying bits of a `double` as a `unint64_t` (common in custom floating-point optimizations or hashing), your had two choices:

1. `reinterpret_cast`: Fast, but often violates **Strict Aliasing Rules.** The compiler might assume the two types don't overlap, leading to optimized code that uses "stale" values.
2. `std::memcpy`: Safe and defined, but syntactically clunky. While modern compilers usually optimize `memcpy` away, it doesn't feel like a cast.

### The Solution: `std::bit_cast`
`std::bit_cast<To>(From)` creates a new object of type `To` by copying the bits of `From`.
```cpp
#include <bit>
#include <cstdint>
#include <iostream>

void quant_example() {
    double price = 42.5;

    auto bits = std::bit_cast<unint64_t>(price);

    if(bits & (1ULL << 63)) {
        // Check sign bit...
    }
}
```
- **Compile Time** (`constexpr`): unlike `reinterpret_cast` or `memcpy`, `std::bit_cast` can be `constexpr`. You can perform bit-level manipulation at compile-time, shifting work from the "Hot Path" to the compilation phase.
- **Requirements**: it only works if `sizeof(To) == sizeof(From)` and both types are Trivially Copyable. The compiler enforces this, preventing you from accidentally casting a 4-byte `int` into an 8-byte `double` (which would cause a buffer over-read).

`std::bit_cast`: 비트 패턴을 재해석함 -> CPU 레지스터 내에서 모든 게 끝날 수 있음.
```cpp
unit64_t i = 0x3FF0000000000000;
double d = std::bit_cast<double>(i); 
```
겉보기에는 새로운 변수가 하나 더 생기는 것 같아 손해처럼 보일 수 있지만, Compiler optimization 단계에서, 컴파일러는 `i`와 `d`를 별개의 메모리 공간으로 만들지 않는다. 그냥 지금 레지스터에 있는 이 비트 뭉치를 이제부터 `double` 연산 명령어에 집어넣어라. 라고 기계어를 짜버린다.

When you use `std:bit_cast`, you aren't changing the bits at all. You are simply moving that exact bit pattern to your other hand (another register) while telling the CPU to treat it as a different type.

In high-performance C++, we categorize casts by whether they operate on **Addresses (Memory)** or **Values (Registers)**. This distinction determines how the CPU handles the data and how the compiler optimizes your code.

| Cast Function | Input (From) | Output (To) | Hardware / Performance Impact |
| :--- | :--- | :--- | :--- |
| **`static_cast<T>(v)`** | Value, Ref, Ptr | Value, Ref, Ptr | **Logical conversion.** May change bits (e.g., int ↔ float). Overhead is **Zero**. |
| **`reinterpret_cast<T>(p)`** | **Address (Ptr/Ref)** | **Address (Ptr/Ref)** | **Re-labels memory.** No bit changes. Risk of **Strict Aliasing** violations (hurts optimization). |
| **`const_cast<T>(p)`** | Address (Ptr/Ref) | Address (Ptr/Ref) | **Qualifier change only.** Strips or adds `const`/`volatile`. No hardware change. |
| **`dynamic_cast<T>(p)`** | Address (Ptr/Ref) | Address (Ptr/Ref) | **Runtime check (RTTI).** Must walk the `vtable`. High latency; **avoid in Hot Paths**. |
| **`std::bit_cast<T>(v)`** | **Pure Value (Reg)** | **Pure Value (Reg)** | **Bit-copy (C++20).** Usually a simple `MOV` between registers. Safest and fastest for type punning. |
