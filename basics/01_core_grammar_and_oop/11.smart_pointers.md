```cpp
    return &a;
}

    int *b = getData();

```

```
5
```

That is because the function variable is only valid during the execution of the function's scope (the lifetime of its stack frame).

Once `getData()` returns, its stack frame is deallocated (popped). The memory address where `a` used to live is now "unprotected" and available for the OS or other functions to overwrite.

```cpp
#include <iostream>
using namespace std;

int* getData(){
    int* a = new int(5);
    return a;
}

int main(){
    int *b = getData();
    cout << *b << "\n";
    cout << *b << "\n";
    cout << *b << "\n";
    delete b;
    
    return 0;
}
```
This might be better than what we had, but we have something called "the smart pointer".

Before talking about the smart pointer, we need to know what a "reference count" is.

The reference count in the heap is a simple and fundamental memory management technique used to track the number of active pointers, references, or handles that point to a specific object or block of memory.

The reference count is typically stored as a hidden integer field within the object's memory block in the heap itself, or in an associated data structure, to ensure good memory locality.

- Mechanism: The count is automatically updated by the language runtime whenever a reference is created, copied, modified, or destroyed:
    - Increment: The count increases when a new pointer or variable is made to point to the object (e.g., in an assignment or function call).
    - Decrement: The count decreases when a reference is removed, goes out of scope, or is reassigned.


```cpp
#include <iostream>
#include <memory> // need this to implement the smart pointer
using namespace std;

shared_ptr<int> getData(){
    shared_ptr<int> a = make_shared<int>(5);
    return a;
}

int main(){
    shared_ptr<int> b = getData();
    cout << *b << "\n"; //5
    cout << *b << "\n"; //5
    cout << *b << "\n"; //5
    
    return 0;
}
```

**1. The internal structure: "The Code Block"**
When you call `make_shared<int>(5)`, the CPU does two things in a single allocation (which is why `make_shared` is faster than `new`):

    1. It allocates memory for the **Integer(5)**.
    2. It allocates a **Control Block** which contains a **Reference Count**.

**2. LifeTime Trace**

Inside `getData()`
    
- `shared_ptr<int> a` is created on the stack.
- The **Reference Count** in the heap becomes **1**.
- When the function reaches `retrun a;`, C++ performs a **copy/move** of the smart pointer to the caller `main`.

The Transition
- For a brief moment, as the value is returned, both the local variable `a` and the receiving variable `b` in `main` might point to the same data.
- The **Reference Counter** handles this perfectly. Even when `getData()` finishes and its local stack frame is destroyed, the count is still at least 1 because `b` now owns it.

Inside `main()`
- `shared_ptr<int> b` now "owns" the heap memory.
- The **Reference Count** is **1**.
- Because the count is greater than zero, the C++ runtime **will not deallocate** the memory. This is why out `cout` statements now consistently print `5` instead of garbage.

`shared_ptr` (Smart Pointer)가 안전한 이유
b와 c는 똑같이 힙 메모리를 쓰지만, 내부적으로 **참조 횟수(Reference Count)**를 관리합니다.

`shared_ptr<int> b = getData();` → Reference Count = 1

`auto c = b;` → b를 c에 복사했으므로 Reference Count = 2

`main` 함수 종료 직전:

- `c`가 스택에서 사라지면서 Count = 1로 감소.
- `b`가 스택에서 사라지면서 Count = 0으로 감소.
- Count가 0이 되는 순간, shared_ptr 내부의 Destructor가 자동으로 `delete`를 실행.

A `weak_ptr` is a smart pointer that holds a **non-owning reference** to an object managed by `shared_ptr`. It is primarily used to break **circular references**, which cause memory leaks. Unlike `shared_ptr`, it does not increment the reference count. To access the underlying data, you must call `.lock()` to temporarily convert it back into a `shared_ptr`, ensuring the object still exists before use.

Traits of `weak_ptr`
- Reference Count: It does not increment the strong reference count. It tracks the object without "owning" it.
- Life-Cycle: The object's lifetime is strictly controlled by `shared_ptr` owners. Even if there are 100 `weak_ptr`s looking at an object, the memory is **immediately deallocated** as soon as the last `shared_ptr` goes out of scope.
- Safety (The `expired()` check): One of the biggest risks with raw pointers is not knowing if the memory they point to is still valid. `weak_ptr` provides the `expired()` method, which allows you to safely check if the object has already been deleted before you try to access it.

### Modern correctness notes (important for interviews)

### 1) `unique_ptr` first, then `shared_ptr` only if needed
- Default ownership model: one owner -> `std::unique_ptr`.
- Promote to `std::shared_ptr` only when lifetime truly has multiple owners.

### 2) Control block details
- `shared_ptr` keeps strong count (owners) and weak count (observers/control lifetime).
- `make_shared` generally performs one allocation for object + control block.
- Constructing `shared_ptr` directly from `new` usually performs separate allocations.

### 3) Threading nuance
- Reference count operations are thread-safe.
- The pointee object is not automatically thread-safe.
- If multiple threads mutate pointee state, still use mutex/atomic as needed.

### 4) Circular reference leak pattern
Two objects owning each other with `shared_ptr` never reach count zero.
Break one direction with `weak_ptr`.

```cpp
struct Node {
    std::shared_ptr<Node> next;
    std::weak_ptr<Node> prev; // weak breaks cycle
};
```

### 5) Passing smart pointers to functions
- Use raw/reference for non-owning access:
  - `void f(const T&);`
  - `void g(T*);` (nullable observer)
- Use `std::unique_ptr<T>` by value for ownership transfer.
- Use `std::shared_ptr<T>` by value only when function should share ownership.

### Full source (11.smart_pointers.cpp)

```cpp
#include <iostream>
#include <memory> // need this to implement the smart pointer
using namespace std;

shared_ptr<int> getData(){
    shared_ptr<int> a = make_shared<int>(5);
    return a;
}

int* getDataPtr(){
    int* a = new int(5);
    return a;
}

unique_ptr<int> getDataUnique(){
    unique_ptr<int> a = make_unique<int>(6);
    return a;
}

int main(){ 
    shared_ptr<int> b = getData();
    auto c = b;
    cout << *b << "\n";
    cout << *c << "\n";
    cout << *b << "\n";

    cout << c.use_count() << "\n"; //2 -> how many are referencing

    int* d = getDataPtr();
    cout << *d << "\n";
    delete d;
    /**
     * If 'delete d;' is not called, the memory stays allocated in the Heap 
     * even after 'd' disappears from the Main Stack.
     * * 1. In a 24/7 Trading Engine, this causes a "Memory Leak."
     * 2. The Heap usage grows indefinitely (Memory Footprint increases).
     * 3. Eventually, the OS triggers an OOM (Out-of-Memory) crash, 
     * killing the entire process.
     */

    unique_ptr<int> e = getDataUnique();
    // unique_ptr f = e; can assign only one reference
    cout << *e << "\n";
    // but you can move the ownership to another ptr
    unique_ptr<int> f = move(e);
    // undefined behavior -> cout << *e << "\n";
    cout << *f << "\n";

    //weak ptr
    shared_ptr<int> sptr = make_shared<int>(10);
    weak_ptr<int> wptr = sptr;

    // ✅ The Recommended Approach (Thread-safe & Atomic)
    if(auto shared = wptr.lock()){ 
        // (1) Atomically checks for existence and increments the Reference Count.
        // (2) As long as 'shared' (a strong shared_ptr) exists in this scope, 
        // the memory cannot be deallocated even if other threads reset the original sptr.
        cout << *shared << "\n";
    } else {
        cout << "Object has already been deallocated" << "\n";
    }

    // ❌ The "Check-then-Act" Approach (Risk of Race Condition)
    if(!wptr.expired()){     
        // (1) Check: "Is the object still alive?"
        // <--- [CRITICAL WINDOW: Another thread could call sptr.reset() right here!]
        
        auto shared = wptr.lock(); // (2) Potential failure: object might have vanished by now.
        if(shared) {
            cout << *shared << "\n"; // (3) Risk of dereferencing a nullptr if not checked again.
        }
    } else {
        cout << "Object has already been deallocated" << "\n";
    }
    
    return 0;
}
```
