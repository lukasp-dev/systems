### Full source (05.encapsulation.cpp)

```cpp
#include <iostream>
using namespace std;

class User {
string first_name;
string last_name;
string ringStatus;

public:
    void setRingStatus(string ringStatus){
        if(ringStatus == "Gold" || ringStatus == "Silver" || ringStatus == "Bronze"){
            this->ringStatus = ringStatus;
        }else{
            this->ringStatus = "No Status";
        }
    }

    string getRingStatus(){
        return this->ringStatus;
    }
};

int main() {
    User user;
    user.setRingStatus("Taco");
    cout << user.getRingStatus() << "\n"; // No Status
    return 0;
}
```
When you write `User("1", "Jewook", "Park")` inside the parentheses, you are creating what is known as a Prvalue (Pure R-value) or a Temporary Object.
- Scope: This object's lifetime is restricted to that single line of code (the statement). 
- Action: 
    1. The compiler allocates memory on the Stack for this temporary `User`.
    2. It calls the constructor.
    3. It then passes this temporary object to `emplace_back`.
    4. Because `User` is passed as an object, the `vector` must either Copy or Move it into its own internal Heap memory.
    5. As soon as the line ends (at the `;`), the temporary object on the Stack is no longer needed, **Destructor #1 fires**.

#### `push_back` vs. `emplace_back`: The Technical Distinction
In your current code, emplace_back is behaving exactly like push_back. To see the performance gain, you must change how you call it.

**Case A:** `push_back(T&& obj)`
push_back always expects an object.

If you give it a temporary, it moves/copies it.

**Cost:** 1 Construction (Stack) + 1 Copy/Move (to Heap) + 1 Destruction (Stack).

**Case B:** `emplace_back(Args&&... args)`
emplace_back uses Variadic Templates and Perfect Forwarding. It doesn't want the object; **it wants the constructor arguments.**

Action: It forwards the strings "1", "Jewook", "Park" directly to the allocated memory in the Heap.

**Mechanism:** It uses Placement New to construct the object in-place.

**Cost:** 1 Construction (directly on Heap). No temporary, no extra destructor.

## OOP interview core (must-know)

### 1) class vs struct
- In C++, `class` and `struct` are almost the same language feature.
- The practical default difference:
  - `struct`: default member access is `public`
  - `class`: default member access is `private`
- Interview answer: use whichever communicates intent better.

### 2) Encapsulation and access control
- `private`: implementation details and invariants.
- `public`: stable API.
- `protected`: visible to derived classes (use carefully).
- Strong pattern: keep data private, expose behavior via methods.

```cpp
class BankAccount {
public:
    explicit BankAccount(int initial) : balance_(initial) {}
    void deposit(int amount) { balance_ += amount; }
    bool withdraw(int amount) {
        if (amount > balance_) return false;
        balance_ -= amount;
        return true;
    }
    int balance() const { return balance_; }
private:
    int balance_;
};
```

### 3) Special member functions and Rule of 0/3/5
- Rule of 0: if you only use RAII members (`std::string`, `std::vector`, `std::unique_ptr`), usually define none.
- Rule of 3: if you define one of destructor/copy ctor/copy assignment, likely need all 3.
- Rule of 5: in modern C++, include move ctor and move assignment too.

```cpp
class Buffer {
public:
    explicit Buffer(std::size_t n) : size_(n), data_(std::make_unique<int[]>(n)) {}
    ~Buffer() = default;

    Buffer(const Buffer& other) : Buffer(other.size_) {
        std::copy(other.data_.get(), other.data_.get() + size_, data_.get());
    }
    Buffer& operator=(const Buffer& other) {
        if (this == &other) return *this;
        Buffer temp(other);
        swap(temp);
        return *this;
    }
    Buffer(Buffer&&) noexcept = default;
    Buffer& operator=(Buffer&&) noexcept = default;

    void swap(Buffer& rhs) noexcept {
        std::swap(size_, rhs.size_);
        data_.swap(rhs.data_);
    }

private:
    std::size_t size_{0};
    std::unique_ptr<int[]> data_{};
};
```

### 4) Inheritance and polymorphism
- Inheritance models "is-a".
- Composition models "has-a". Prefer composition by default.
- Runtime polymorphism requires:
  - base pointer/reference
  - `virtual` function in base
  - overriding function in derived

```cpp
class Shape {
public:
    virtual ~Shape() = default;          // critical for polymorphic delete
    virtual double area() const = 0;     // pure virtual = abstract base
};

class Circle final : public Shape {
public:
    explicit Circle(double r) : r_(r) {}
    double area() const override { return 3.1415926535 * r_ * r_; }
private:
    double r_;
};
```

### 5) virtual destructor: mandatory rule
- If a class has any virtual function, destructor should almost always be virtual.
- Otherwise deleting `Derived` via `Base*` is undefined behavior.

### 6) override/final best practice
- Always put `override` on overridden methods.
- Use `final` when further override/inheritance should be blocked.
- This catches signature mismatches at compile time.

### 7) Object slicing
- Passing/storing derived object by value into base type slices derived part.

```cpp
struct Base { virtual ~Base() = default; virtual void f() const {} };
struct Derived : Base { void f() const override {} int extra{}; };

void takesBase(Base b) { b.f(); } // slicing
```

- Fix: pass `Base&` or `const Base&`, or use `std::unique_ptr<Base>`.

### 8) Static vs dynamic dispatch
- Non-virtual call: compile-time binding (fast, inlinable).
- Virtual call: runtime dispatch via vtable (small indirection cost).
- Interview framing: choose virtual only when you need runtime substitution.

### 9) Interface-style design in C++
- Use abstract base classes for runtime plugin boundaries.
- Use templates/concepts for compile-time polymorphism when types are known.
- Modern C++ often mixes both: runtime architecture + compile-time hot paths.

### 10) Common OOP interview pitfalls
1. Missing virtual destructor in polymorphic base.
2. Forgetting `override`, accidentally overloading instead.
3. Returning owning raw pointers.
4. Using inheritance when composition is simpler.
5. Ignoring copy/move semantics in resource-owning classes.

### 11) End-to-end polymorphism design example (interface + ownership + factory)

This is the full interview pattern: abstract interface, concrete implementations, ownership via `unique_ptr`, and runtime dispatch.

```cpp
#include <iostream>
#include <memory>
#include <string>
#include <vector>

class Notifier {
public:
    virtual ~Notifier() = default;
    virtual void send(std::string_view msg) = 0;
};

class EmailNotifier final : public Notifier {
public:
    void send(std::string_view msg) override {
        std::cout << "[EMAIL] " << msg << "\n";
    }
};

class SmsNotifier final : public Notifier {
public:
    void send(std::string_view msg) override {
        std::cout << "[SMS] " << msg << "\n";
    }
};

enum class Channel { Email, Sms };

std::unique_ptr<Notifier> make_notifier(Channel c) {
    switch (c) {
        case Channel::Email: return std::make_unique<EmailNotifier>();
        case Channel::Sms:   return std::make_unique<SmsNotifier>();
    }
    return std::make_unique<EmailNotifier>();
}

int main() {
    std::vector<std::unique_ptr<Notifier>> pipeline;
    pipeline.push_back(make_notifier(Channel::Email));
    pipeline.push_back(make_notifier(Channel::Sms));

    for (auto& n : pipeline) n->send("system healthy");
}
```

Why this is a strong design:
1. interface boundary is explicit (`Notifier`)
2. polymorphic deletion is safe (`virtual ~Notifier`)
3. ownership is explicit (`unique_ptr`)
4. caller code depends on abstraction, not concrete class
