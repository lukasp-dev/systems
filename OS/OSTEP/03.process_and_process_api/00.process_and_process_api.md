## 1. One-page cheat sheet

> **Core mental model**
> `fork()` = duplicate the current process. `exec()` = transform the current
> process into a new program. `wait()` = make a parent process wait for a child
> process to finish.

| Topic | Meaning | Remember this |
| --- | --- | --- |
| Process | A running program plus its resources and OS state. | Not just code; it includes memory, registers, files, and OS bookkeeping. |
| Mechanism | How something is done. | Example: how to context switch. |
| Policy | Which choice the OS makes. | Example: which process should run next. |
| Ready | Can run, but not currently on CPU. | Waiting for CPU. |
| Running | Currently executing on CPU. | Actually using the CPU right now. |
| Blocked | Cannot run until an event happens. | Usually waiting for I/O. |
| PCB / Process descriptor | OS record for one process. | In xv6, this is `struct proc`. |
| Process list / table | Collection of PCBs/descriptors. | One entry per process. |
| `fork()` | Creates a child copy of the current process. | Child gets `rc = 0`; parent gets child PID. |
| `wait()` | Parent blocks until child exits. | Provides ordering/synchronization. |
| `exec()` | Replaces current process image with a new program. | Same PID; new code/data/stack/heap; no return on success. |
| Redirection | Change file descriptors before `exec()`. | Open FDs stay open across `exec()`. |

## 2. What is a process?

A process is the OS abstraction for a running program. It gives the program the
illusion that it has its own CPU and its own private memory, even though the OS
is multiplexing real hardware among many processes.

A useful formula:

```
Process = program code + memory state + CPU register state + OS bookkeeping
```

The program file on disk is passive. A process is active: it has a current
instruction, a stack, a heap, open files, and a place in the OS scheduler.

---

## 3. Mechanism vs. policy

OSTEP emphasizes that operating systems often separate low-level **mechanisms**
from high-level **policies**.

| Concept | Question answered | Example |
| --- | --- | --- |
| Mechanism | How does the OS do something? | How does the OS perform a context switch? |
| Policy | Which choice should the OS make? | Which process should the OS run right now? |

> **Why it matters**
> If mechanism and policy are separated, the OS can change scheduling policies
> without rewriting the context-switching mechanism. This is modularity.

## 4. How the OS prepares a process

When a program starts, the OS prepares an address space and other resources.
Modern OSes often do this lazily, not eagerly.

| Style | Meaning | Example |
| --- | --- | --- |
| Eager | Do all work immediately. | Load all code and data into physical memory before the process runs. |
| Lazy | Do only what is needed now; delay the rest. | Set up mappings now, but load pages only when the process touches them. |

Before running the process, the OS prepares:

- **Code/text region:** machine instructions of the program.
- **Static data region:** global/static variables.
- **Runtime stack:** function calls, local variables, return addresses, `argc`/`argv`.
- **Heap:** dynamic memory used by `malloc`/`new`.
- **I/O state:** standard file descriptors and other open files.
- **Initial CPU/register state:** where execution begins.

```
High addresses
+------------------+
| Stack            |  grows downward; calls/local variables
|                  |
|                  |
| Heap             |  grows upward; malloc/new
+------------------+
| Static data      |  globals/static variables
+------------------+
| Code/Text        |  program instructions
+------------------+
Low addresses
```

## 5. Process states

The simple OSTEP model has three main states: **Running**, **Ready**, and **Blocked**.

| State | Meaning | Transition example |
| --- | --- | --- |
| Running | Currently using the CPU. | Chosen by scheduler from Ready. |
| Ready | Ready to run, but waiting for CPU. | Timer interrupt moves Running to Ready. |
| Blocked | Cannot run until something happens. | Process waits for disk/input/network I/O. |

```
Ready   -> Running   scheduler chooses it
Running -> Ready     timer interrupt / preemption
Running -> Blocked   process waits for I/O or event
Blocked -> Ready     I/O or event completes
```

> **Key fact: timer interrupt / preemption**
> The OS programs a hardware **timer** to fire at fixed intervals (a *tick*).
> Each tick raises a **timer interrupt** that forcibly transfers control from the
> running process back to the kernel. The scheduler can then **preempt** the
> current process \u2014 move it from **Running** back to **Ready** and pick another
> one to run. This is what guarantees that a single CPU-bound process cannot hog
> the CPU forever; the OS always regains control. Preemption is a *mechanism*;
> deciding which process runs next is *policy*.

### xv6 process states

| xv6 state | Meaning | Simple OSTEP equivalent |
| --- | --- | --- |
| UNUSED | Free slot in process table. | None |
| EMBRYO | Process is being created. | Temporary creation state |
| SLEEPING | Waiting on an event/channel. | Blocked |
| RUNNABLE | Ready to run. | Ready |
| RUNNING | Currently executing. | Running |
| ZOMBIE | Exited but parent has not collected status yet. | Terminated but not fully cleaned |

> **Common confusion**
> In xv6, `RUNNABLE` means ready to run, not currently running. `RUNNING` means
> actually on the CPU.

## 6. Process data structures: process list, PCB, descriptor

The OS needs to remember every process, even when it is not running. It does
this with process-related data structures.

![Process data structures: process list, PCB, and descriptor](https://github.com/user-attachments/assets/a6d0d875-81ed-49d8-9a7e-30fabe1f7b9c)

| Term | Meaning | Relationship |
| --- | --- | --- |
| Process descriptor | The OS record for one process. | Same general idea as a PCB. |
| PCB | Process Control Block: bookkeeping information for one process. | In xv6, `struct proc` plays this role. |
| Process list/table | The OS collection of process descriptors/PCBs. | One entry per process. |

```
Process table / process list
    +-- PCB / descriptor for process 1
    +-- PCB / descriptor for process 2
    +-- PCB / descriptor for process 3
```

### xv6 `struct context`

The context stores CPU registers that xv6 saves and restores to stop and later
restart a process.

```c
struct context {
    int eip;  // instruction pointer: where to continue
    int esp;  // stack pointer
    int ebx;
    int ecx;
    int edx;
    int esi;
    int edi;
    int ebp;  // base pointer
};
```

### xv6 `struct proc`

`struct proc` is xv6's process descriptor / PCB. It records memory, state,
identity, parent, open files, current directory, saved context, and trap
information.

```c
struct proc {
    char *mem;                  // Start of process memory
    uint sz;                    // Size of process memory
    char *kstack;               // Bottom of kernel stack for this process
    enum proc_state state;      // Process state
    int pid;                    // Process id
    struct proc *parent;        // Parent process
    void *chan;                 // If nonzero, sleeping on chan
    int killed;                 // If nonzero, has been killed
    struct file *ofile[NOFILE]; // Open files
    struct inode *cwd;          // Current directory
    struct context context;     // Switch here to run process
    struct trapframe *tf;       // Current interrupt/trap/syscall state
};
```

| Field | Meaning |
| --- | --- |
| `pid` | Unique process ID. |
| `state` | Current state such as RUNNABLE, RUNNING, SLEEPING, ZOMBIE. |
| `mem`, `sz` | Process memory location and size in this xv6 model. |
| `kstack` | Kernel stack used when this process is running in the kernel. |
| `parent` | Pointer to the process that created it. |
| `chan` | The event/channel the process is sleeping on. |
| `killed` | Flag saying the process has been marked for termination. |
| `ofile[]` | Open files belonging to the process. |
| `cwd` | Current working directory. |
| `context` | Saved registers for context switching. |
| `trapframe` | Saved user state when entering kernel through interrupt/trap/system call. |

> **context vs. trapframe**
> `context` is mainly for context switching between processes. `trapframe` saves
> the user-level state at the moment of a system call, interrupt, or trap.

> **PCB vs. file descriptor (don't confuse them)**
> A file descriptor is **not** the same as the PCB \u2014 it is *contained inside*
> it. The PCB (`struct proc`) is the single record for the whole process; the
> `ofile[]` field is that process's **file-descriptor table**, where each slot is
> one open file (`ofile[0]`=stdin, `ofile[1]`=stdout, `ofile[2]`=stderr). So the
> relationship is a containment hierarchy:
> `PCB \u2283 FD table (ofile[]) \u2283 individual FD`.
> This is why `exec()` keeps the same PCB *and* its open FDs (except those marked
> close-on-exec): the FD table lives inside the PCB, which is never recreated.

> **See also**
> The full mechanism of how the OS saves one process and restores another (the
> trap frame, the scheduler, and `return-from-trap`) is covered in
> [04.context_switch/00.context_switch.md](../04.context_switch/00.context_switch.md).

## 7. The Process API

The Process API is the set of operations the OS provides so programs and shells
can manage processes.

| Category | Question it answers | Examples |
| --- | --- | --- |
| Create | How do we create a new process? | `fork()`, then often `exec()` |
| Destroy | How do we terminate a process? | `exit()`, `kill()`, signals |
| Wait | How do we wait for a process to finish? | `wait()`, `waitpid()` |
| Miscellaneous control | How do we control process behavior? | signals, priority, stop/continue |
| Status | How do we inspect process information? | `getpid()`, wait status, process state tools |

## 8. fork(): duplicating a process

`fork()` creates a new child process by duplicating the currently running
process. The key point: `main()` is not called again. The process is cloned
mid-execution.

```c
printf("hello (pid: %d)\n", (int)getpid());
int rc = fork();   // one call, but two processes return from it
```

| Process | `rc` value after `fork()` | Meaning |
| --- | --- | --- |
| Child | `0` | This process is the newly created child. |
| Parent | child's PID | This process is the original parent. |
| Error case | negative value | fork failed; no child was created. |

```c
if (rc < 0) {
    fprintf(stderr, "fork failed\n");
    exit(1);
} else if (rc == 0) {
    printf("child (pid: %d)\n", (int)getpid());
} else {
    printf("parent of %d (pid: %d)\n", rc, (int)getpid());
}
```

> **Why output order can change**
> After `fork()`, parent and child are separate runnable processes. The
> scheduler decides which one runs first, so output order is nondeterministic
> unless the program synchronizes with `wait()`.

---

## 9. wait(): synchronizing parent and child

`wait(NULL)` makes the parent block until one of its children exits. This
controls ordering and lets the parent collect the child's exit status.

```c
int rc_wait = wait(NULL);
printf("parent of %d (rc_wait: %d) (pid: %d)\n",
       rc, rc_wait, (int)getpid());
```

In the common case with one child, `wait()` returns the PID of the child that
finished. In the examples we studied, that is why `rc_wait` often equals `rc`.

```
Without wait():
    hello
    parent or child may print next; order is nondeterministic

With wait():
    hello
    child
    parent prints after child exits
```

## 10. exec(): replacing the current program

`exec()` does not create a new process. It replaces the current process image
with a new program.

> **Most important exec idea**
> On successful `exec()`, the process keeps its PID, but its old code, data,
> heap, and stack are replaced by the new program. Code after `exec()` only runs
> if `exec()` fails.

```c
char *myargs[3];
myargs[0] = strdup("wc");
myargs[1] = strdup("figure5.3.c");
myargs[2] = NULL;
execvp(myargs[0], myargs);
printf("this should not print out");  // only runs if execvp fails
```

| Before execvp | After successful execvp |
| --- | --- |
| Child is running your C program. | Same child process is now running `wc`. |
| PID is the child's PID. | PID is still the same child PID. |
| Code after exec exists in old program. | Old program is gone, so code after exec does not run. |
| Open file descriptors exist. | Open file descriptors remain open unless marked close-on-exec. |

## 11. fork + exec + wait: how a shell runs commands

The three calls combine into the classic Unix shell pattern.

```
Parent shell:
    fork()
    if parent:
        wait()        // wait for child command to finish

Child:
    optional setup    // redirection, pipes, environment, directory, etc.
    exec(command)     // replace child with requested program
```

This design is powerful because `fork()` gives the child a short setup window
before `exec()`. The child can change file descriptors or other process state,
and then the new program inherits that setup.

## 12. Redirection and file descriptors

Unix processes usually start with three standard file descriptors.

| FD number | Name | Normal meaning |
| --- | --- | --- |
| 0 | stdin | standard input, often keyboard/terminal input |
| 1 | stdout | standard output, often terminal output |
| 2 | stderr | standard error, often terminal error output |

### Redirection example

The redirection example shows the benefit of separating `fork()` and `exec()`.
The child changes stdout before it becomes `wc`.

```c
int rc = fork();
if (rc == 0) {
    close(STDOUT_FILENO);  // close fd 1
    open("./p4.output", O_CREAT | O_WRONLY | O_TRUNC, S_IRWXU);

    char *myargs[3];
    myargs[0] = strdup("wc");
    myargs[1] = strdup("figure5.4.c");
    myargs[2] = NULL;
    execvp(myargs[0], myargs);
} else {
    wait(NULL);
}
```

Why this redirects output:

1. `STDOUT_FILENO` is file descriptor 1.
2. The child closes fd 1.
3. `open()` returns the lowest available file descriptor, so `p4.output` usually becomes fd 1.
4. Open file descriptors stay open across `exec()`.
5. When `wc` writes to stdout, stdout now points to `p4.output` instead of the screen.

> **Shell equivalent**
> This is how a shell can implement: `wc figure5.4.c > p4.output`

A more explicit production-style version often uses `dup2()`:

```c
int fd = open("./p4.output", O_CREAT | O_WRONLY | O_TRUNC, S_IRWXU);
dup2(fd, STDOUT_FILENO);
close(fd);
execvp(myargs[0], myargs);
```

## 13. Process control, signals, users, and superuser

Process control is available through **signals**. Signals are a way to notify or
control a process.

| Signal/action | Meaning |
| --- | --- |
| Stop | Suspend a process so it does not run. |
| Continue | Resume a stopped process. |
| Terminate | Ask or force a process to end. |
| Kill | Often used informally to mean sending a terminating signal. |

The OS also uses the concept of **users** to decide which processes a person can
control.

- A normal user can usually control only their own processes.
- A user should not be able to kill or interfere with another user's processes.
- The superuser/root can control all processes and do many privileged actions.
- Superuser privileges should be used infrequently and cautiously because
  mistakes or attacks can damage the whole system.

## 14. Process exit behavior: zombies, orphans, and reparenting

When a process calls `exit()`, it does not vanish instantly. The OS must keep a
small amount of bookkeeping around until the parent collects the exit status,
and it must decide what happens to any children the exiting process leaves
behind. Two situations matter: **zombies** (a child exits before the parent
waits) and **orphans** (a parent exits before its children).

| Term | Who exited first | What happens |
| --- | --- | --- |
| Zombie | Child exits, parent has **not** called `wait()` yet. | Child is dead but its descriptor lingers so the parent can still read the exit status. |
| Orphan | Parent exits while a child is **still running**. | Child loses its parent and must be **reparented** to a new one. |

### Zombies

A **zombie** is a process that has finished executing but whose exit status has
not yet been collected by its parent via `wait()`. The kernel keeps just enough
state (PID + exit status) so the parent can reap it later. This is exactly the
xv6 `ZOMBIE` state from the [xv6 process states table](#xv6-process-states):
*"Exited but parent has not collected status yet."*

- The zombie holds almost no resources — only its entry in the process table.
- Once the parent calls `wait()`/`waitpid()`, the kernel frees that entry and
  the PID is reusable. This is called **reaping**.
- If a parent never calls `wait()`, zombies pile up and leak PID-table slots.

> **Why zombies exist**
> Every process needs a parent to read its exit status. The OS cannot throw the
> status away immediately, because the parent might still want it. So a finished
> child stays a zombie until reaped.

### Orphans and reparenting

When a parent exits while its children are still alive, those children become
**orphans**. Because every process must have a living parent to eventually reap
it, the kernel **reparents** each orphan to a new parent. There are two models:

| Model | New parent of the orphan | Era / where it's used |
| --- | --- | --- |
| Reparent to `init` | **PID 1 (`init`)** — orphans flatten all the way to the top. | Classic Unix default. |
| Reparent to nearest **subreaper** | The closest ancestor that registered as a subreaper. | Modern Linux (since 3.4): `systemd`, Docker/containerd. |

#### Reparent to `init` (classic)

All orphaned descendants jump straight to **PID 1**, no matter how deep they
were in the tree. `init` runs a loop that continuously calls `wait()`, so it
reaps any orphan that later exits, preventing permanent zombies.

#### Subreapers (modern)

A **subreaper** is a process that volunteers to act as a stand-in `init` for its
own subtree. It marks itself with:

```c
prctl(PR_SET_CHILD_SUBREAPER, 1);
```

After that, when any of its descendants is orphaned, the kernel reparents the
orphan to the **nearest ancestor subreaper** instead of PID 1. This keeps
orphans grouped under the manager that owns them — `systemd` uses it so a
service's stray children stay under that service, and container runtimes use it
so the container's init can track and reap every process inside.

```
Reparent to init (classic)          Reparent to subreaper (-R style)

  a (exits)                            a (exits, S above is subreaper)
  ├── b   ──┐                          S
  └── c     │ b, c, d, e               └── b   (b, c stay under a's parent;
      └── d │ all jump to                  └── c    deeper kids stay local)
          └── e  PID 1 / root
```

### Tie-in: the `fork.py` simulator

The OSTEP `fork.py` homework models both policies directly:

| `fork.py` behavior | Real Linux equivalent |
| --- | --- |
| **Default** — orphans reparent to the root process. | Classic: reparent to `init` (PID 1). |
| **`-R` (`--local_reparent`)** — orphans reparent to the exiting process's parent. | Modern: reparent to the nearest **subreaper**. |

So running `python fork.py -A a+b,b+c,c+d,c+e,c-` and watching where `d` and `e`
go when `c` exits is a hands-on demonstration of reparenting: without `-R` they
flatten to root (init model), and with `-R` they stay under `b` (subreaper
model).

> **Final takeaway**
> The OS manages processes by keeping one descriptor/PCB per process, moving

> environment first; this is the foundation of shells, redirection, and pipes.
