# OSTEP — Context Switch Study Guide

> *Based on notes from Operating Systems: Three Easy Pieces (Limited Direct Execution)*

> **Big picture**
> A context switch is the mechanism the OS uses to stop running one process and
> start running another, while keeping the illusion that each process has the CPU
> to itself. It is part of *limited direct execution*, not the Process API.

## 1. Context switch: saving one process and restoring another

A **context switch** is how the OS stops running one process and starts running
another, so that the second process resumes as if it had been running all along
and had just entered the kernel.

> **Key insight**
> A full switch from process A to process B involves **two different save/restore
> layers**: one for A's **user-mode** state and one for A's **kernel-mode**
> state. They happen at different moments and are stored in different places.

---

## 2. What is a timer interrupt?

A **timer interrupt** is a periodic hardware signal, fired by a programmable
timer chip, that forces the CPU to stop the running process and jump into the
OS. It is the mechanism that lets the OS **regain control of the CPU** from a
process that would otherwise run forever.

> **Why it exists**
> Under *limited direct execution*, a user process runs directly on the CPU for
> speed. But if it never gives the CPU back (an infinite loop, or a long
> computation), the OS would be stuck. The timer interrupt guarantees control
> returns to the OS on a fixed schedule, no matter what the process is doing.

How it is set up and fires:

```
At boot:
    OS programs the timer to fire every N ms (the "tick", e.g. 1-10 ms)
    OS installs an interrupt handler (on x86, an entry in the IDT)

While a process runs:
    timer counts down -> fires -> CPU switches user -> kernel mode
    CPU jumps to the OS's preset handler
    OS handles the tick (update time, maybe call the scheduler)
```

Key properties:

| Property | Meaning |
| --- | --- |
| Periodic | Fires on a fixed schedule (the tick rate), unlike event-driven I/O interrupts. |
| Asynchronous | Can interrupt the process at any instruction; the process did not ask for it. |
| Forces a mode switch | Always moves the CPU from user mode into kernel mode. |
| Enables preemption | It is what makes **preemptive** multitasking possible — the OS can take the CPU away. |

> **Interrupt is not the same as a context switch**
> The timer interrupt only *returns control to the OS*. Most ticks just update
> bookkeeping and resume the **same** process. A context switch happens only if
> the scheduler decides another process should run (see the next sections).

---

## 3. Timer interrupt: saving user-mode state

The CPU is executing A's code in **user mode**. Then a timer interrupt fires and
the CPU switches from user mode into kernel mode. To make it possible to return
to A later, the CPU/OS saves A's user-mode state onto **A's kernel stack**.

This saved user-mode snapshot is called a **trap frame**, and contains roughly:

| Saved item | Why it is needed |
| --- | --- |
| User program counter (PC) | The exact instruction A was about to run. |
| User stack pointer | A's user-level stack position. |
| User registers | A's in-progress computation. |
| Flags / condition codes | A's processor status. |

Sequence so far:

```
A running in user mode
        |
timer interrupt fires
        v
CPU switches user -> kernel mode
        |
CPU jumps into the OS trap handler
        |
A's user state saved on A's kernel stack (trap frame)
```

> **Whose behalf?**
> At this point the OS is running, but it is running **on behalf of A**: the CPU
> is in kernel mode, using **A's kernel stack**.

---

## 4. The scheduler decides whether to switch

The timer interrupt handler may eventually call the **scheduler**, which asks the
policy question:

> Should we keep running A, or should we run another process B?

If the scheduler chooses to keep A, the OS simply returns from the trap and A
continues. If the scheduler chooses **B**, the OS performs the actual context
switch — the **second** save/restore.

---

## 5. Context switch: saving kernel-mode state

A is no longer running in user mode; A is already **inside the kernel** because
of the timer interrupt. So now the OS must save A's **kernel-mode** state and
restore B's. This state lives in each process's descriptor (e.g. `struct proc` /
`struct context`), not on the kernel stack.

| Saved/restored item | Where it lives |
| --- | --- |
| Kernel stack pointer | Process structure (`struct context`). |
| Kernel registers | Process structure (`struct context`). |
| Return address inside the kernel | Process structure (`struct context`). |

The switch looks like:

```
save A's kernel registers / context
save A's kernel stack pointer
        |
load B's kernel stack pointer
restore B's kernel registers / context
```

> **The magical part**
> After this, the CPU is no longer using A's kernel stack — it is using **B's
> kernel stack**. The OS has moved from "running in kernel mode on A's kernel
> stack" to "running in kernel mode on B's kernel stack."

The context switch does **not** jump directly from A's user code to B's user
code. Instead, it changes the **kernel execution context** from A to B. It now
looks as if B had previously trapped into the kernel and the OS is finishing B's
trap/interrupt handler. This is why OSTEP says the system changes from "as if it
had trapped from A" to "as if it had trapped from B."

---

## 6. Return-from-trap resumes B

Eventually the OS finishes handling the interrupt/trap and executes a special
return-from-trap instruction (on x86, `iret` / `iretq`). This instruction tells
the CPU to:

1. Leave kernel mode.
2. Restore the saved user state (the trap frame).
3. Resume user-mode execution.

But after the context switch the CPU is using **B's kernel stack**, so the trap
frame restored is **B's**. Therefore the CPU resumes **B** in user mode.

```
A running in user mode
        |
timer interrupt
        v
A's user state saved on A's kernel stack (trap frame)
        |
OS scheduler runs
        |
context switch: save A's kernel context, restore B's kernel context
        |
now running on B's kernel stack
        |
return-from-trap
        |
B resumes in user mode
```

## 7. Why there are two saves

| | First save (user) | Second save (kernel) |
| --- | --- | --- |
| When | Timer interrupt / trap occurs. | OS voluntarily switches A → B inside the kernel. |
| What | A's user registers, PC, user SP, flags. | A's kernel registers, kernel SP, kernel return address. |
| Stored where | A's kernel stack (trap frame). | A's process/thread structure (`struct context`). |
| Purpose | Resume A's interrupted **user** program. | Resume A's **kernel** execution path later. |

> **Save is about A; restore is about B**
> Both saves above are **A's** state, kept so A can resume later. B's state is not
> saved here — it is **restored**, because B saved its own state the same way the
> last time it was switched out.
>
> ```
> leaving A:   save A's user state      (A's kernel stack)
>              save A's kernel context  (A's struct context)
> entering B:  restore B's kernel context (B's struct context)
>              restore B's user state     (B's kernel stack, at return-from-trap)
> ```

Full end-to-end sequence:

1. Trap/interrupt saves A's user-mode state on A's kernel stack.
2. The OS enters the scheduler.
3. The OS saves A's kernel-mode context.
4. The OS restores B's kernel-mode context.
5. The CPU is now running in the kernel on B's kernel stack.
6. `return-from-trap` restores B's user-mode state.
7. B continues running in user mode.

> **정리 (trapframe vs context)**
>
> - **trapframe = user 상태 저장** (PC, user 레지스터, user SP)
>   - 언제? user → kernel 진입할 때 (trap)
>   - 어디에? **커널 스택**에 데이터가 있고, **PCB에는 포인터만** 저장
>     (`struct trapframe *trapframe;`)
> - **context = kernel 상태 저장** (kernel 레지스터, kernel SP, 복귀주소)
>   - 언제? 커널 안에서 프로세스 **A → B 전환**할 때
>   - 어디에? **PCB 안에 구조체로 직접** 박힘 (`struct context context;`)
>
> 그래서 전환은 **2번 저장(two saves)**:
> 1. user → kernel: **trapframe** 저장
> 2. A → B 전환: **context** 저장
>
> 한 줄: **trapframe = 유저 상태(커널 스택, PCB엔 포인터), context = 커널 상태(PCB 안 직접).**

## 8. Review questions

1. During a context switch, which two pieces of state are saved, and where is each one stored?
2. After the context switch restores B's kernel context, why does `return-from-trap` resume B and not A?
3. Why is A's user-mode state saved on A's kernel stack, while A's kernel-mode context is saved in A's process structure?
4. What does the OS mean when it says the system goes from "as if it had trapped from A" to "as if it had trapped from B"?
5. Where does the scheduler's policy decision fit into the context-switch mechanism?
6. Why is a timer interrupt necessary, and how does it differ from a context switch?

> **Final takeaway**
> A context switch is two save/restore steps at two layers: the trap saves user
> state on the kernel stack, and the switch saves kernel state in the process
> structure. After the switch the CPU runs on B's kernel stack, so
> `return-from-trap` naturally resumes B in user mode.
