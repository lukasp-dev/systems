## 1. One-page cheat sheet

| Topic | Meaning | Remember this |
| --- | --- | --- |
| Workload / job | A process running in the system. | What the scheduler is trying to multiplex on the CPU. |
| Turnaround | `T(completion) − T(arrival)` | Primary metric in this chapter; lower is better. |
| Response | `T(first run) − T(arrival)` | Matters for interactive jobs; not optimized by STCF. |
| Preemption | OS forcibly interrupts a running task. | Modern schedulers are preemptive; batch-era schedulers often were not. |
| FCFS / FIFO | Run jobs in arrival order, to completion. | Simple; suffers from **convoy effect**. |
| SJF | Run shortest job first (non-preemptive). | Optimal average turnaround if all jobs arrive together. |
| STCF (SRTF) | Preemptive SJF — run job with shortest **remaining** time. | Better turnaround when jobs arrive over time; can hurt response time. |
| RR | Each ready job gets a fixed **time slice**, then goes to back of queue. | Fair; often bad average turnaround; quantum should be a multiple of the timer-interrupt period. |
| Convoy effect | Short jobs stuck behind one long job. | Classic FCFS failure mode. |
| Context switch cost | More than saving registers. | Caches, TLB, branch predictors — switching jobs flushes hot on-chip state. |

## 2. Workload and assumptions

OSTEP starts with simplified assumptions about the **workload** (the set of
**jobs** / processes in the system):

1. Each job runs for the same amount of time.
2. All jobs arrive at the same time.
3. Once started, each job runs to completion (non-preemptive).
4. All jobs only use the CPU (no I/O).
5. The run time of each job is known.

These are deliberately unrealistic. The chapter relaxes them gradually. The
hardest assumption to drop in practice is (5): the OS cannot see the future.
That gap is what motivates **MLFQ** in the next chapter.

---

## 3. Scheduling metrics

**Turnaround time**

```
T_turnaround = T(completion) − T(arrival)
```

With all jobs arriving at once, `T(arrival) = 0`, so turnaround equals completion time.

**Fairness** (e.g., Jain's fairness index) is another metric. Performance and
fairness often conflict — a policy can optimize turnaround while starving some
jobs.

---

## 4. FCFS / FIFO

**First Come, First Served** — run jobs in arrival order until each finishes.

- Simple and easy to implement.
- Works fine when jobs are similar length.
- **Convoy effect:** a large, CPU-intensive job at the front of the queue blocks
  or delays many smaller jobs waiting behind it.

Example (A=100s, B=10s, C=10s, all arrive at t=0):

```
|-- A (100s) --|-- B --|-- C --|
Average turnaround: (100 + 110 + 120) / 3 = 110
```

---

## 5. SJF (non-preemptive)

**Shortest Job First** — run the shortest job first, then the next shortest.

- Fixes convoy when all jobs are present at once.
- With simultaneous arrival, SJF minimizes average turnaround (among non-preemptive policies in this model).
- **Non-preemptive:** once a job starts, it runs to completion.

Same example with SJF (run B, C, then A):

```
|-- B --|-- C --|-- A (100s) --|
Average turnaround: (10 + 20 + 120) / 3 = 50
```

If jobs arrive at different times, pure SJF still lets an already-running long
job block shorter jobs that arrive later — convoy returns.

---

## 6. STCF / SRTF (preemptive)

**Shortest Time-to-Completion First** (also **Shortest Remaining Time First**) —
preemptive version of SJF.

**Preemption:** the system forcibly interrupts a running task and switches to
another — without waiting for the task to finish or yield voluntarily.

Whenever a new job arrives (or a job becomes ready), run whichever job has the
**shortest remaining** run time.

- Better average turnaround when jobs arrive over time.
- **Response time can suffer:** a long job that keeps getting preempted by
  shorter arriving work may wait a long time for its first slice (or see long
  gaps before first meaningful run, depending on arrival pattern).

---

## 7. Response time

```
T_response = T(first run) − T(arrival)
```

Turnaround cares about **finish**; response cares about **first run** — critical
for interactive workloads (shell, UI).

| Policy | Turnaround | Response |
| --- | --- | --- |
| FCFS | Poor if convoy | OK for first in line |
| SJF / STCF | Strong (when lengths known) | STCF can be **bad** for long jobs when short jobs keep arriving |
| RR | Often poor for long jobs | Good if quantum is small |

---

## 8. Round Robin (preemptive)

**Round Robin** — each ready job gets a **time slice** (scheduling **quantum**),
then goes to the back of the ready queue if not finished.

- **Fair:** every ready job gets equal turns on the CPU.
- **Time slice should be a multiple of the timer-interrupt period.** Timer
  interrupts fire at fixed intervals; if quantum is not aligned, actual slice
  lengths become inconsistent.
- **Trade-off:** fair policies like RR often perform **poorly on turnaround
  time** — long jobs need many quanta, and each preemption adds overhead.

---

## 9. Trade-offs and context-switch cost

OSTEP's chapter summary: two families of approaches, each good where the other
is bad.

| Family | Idea | Strong on | Weak on |
| --- | --- | --- | --- |
| Shortest-remaining-time | STCF / SJF | Turnaround | Response, fairness, needs known run times |
| Equal sharing | Round Robin | Fairness, response | Turnaround, throughput if quantum too small |

**Context switching is expensive beyond register save/restore.** While a program
runs, it builds up state in CPU **caches**, **TLBs**, **branch predictors**, and
other on-chip structures. Switching to another job flushes that state and warms
state for the new job — a noticeable performance cost. Very short RR quanta
can spend more time switching than doing useful work.

The chapter also discusses incorporating **I/O** (CPU vs I/O overlap), but the
fundamental problem remains: without knowing the future, simple policies cannot
simultaneously optimize turnaround, response, and fairness. **MLFQ** uses recent
history to approximate what SJF would do if run times were known — covered in
[01.multi_level_feedback_queue.md](./01.multi_level_feedback_queue.md).

11. Why does OSTEP introduce MLFQ after this chapter?

> **Final takeaway**
> FCFS, SJF, STCF, and RR are teaching policies: each optimizes one metric under
> assumptions the real world violates. Real schedulers (starting with MLFQ) trade
> these ideals for behavior learned from the recent past.
